# Plans

For a repository this small, plans should stay lightweight.

## Use a Checked-In Plan When

- the change touches both structure and content
- the change will span multiple sessions
- the change introduces a new section or architectural boundary

## Plan Format

- objective
- constraints
- files or directories affected
- validation steps
- follow-up cleanup

If a change is simple enough to complete in one pass, prefer updating the docs directly instead of creating plan churn.
